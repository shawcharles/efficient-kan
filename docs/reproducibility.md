# Reproducibility

`efficient-kan` is often used as a dependency inside downstream research
pipelines. Treat the package version, import path, PyTorch version, random
seeds, device, dtype, and adaptive-grid schedule as part of the research
artifact.

## Random Seeds

The package uses PyTorch initialization and PyTorch operations. Set PyTorch
seeds in scripts, examples, tests, and simulation runners:

```python
import torch


torch.manual_seed(123)
```

For CUDA workflows, also consider:

```python
torch.cuda.manual_seed_all(123)
```

Seeding controls random initialization and random tensors generated by PyTorch.
It does not guarantee identical results across all PyTorch versions, devices,
CUDA libraries, BLAS libraries, or hardware.

## PyTorch Version Sensitivity

`efficient-kan` relies on PyTorch for:

- parameter initialization;
- tensor operations;
- autograd;
- `torch.linalg.lstsq` in spline coefficient solves;
- CPU and CUDA kernels;
- dtype and device behavior.

Different PyTorch versions or builds can produce small numerical differences.
These differences are most relevant for least-squares solves, CUDA execution,
float32 calculations, and training loops where tiny changes can accumulate.

For research artifacts, record:

- `efficient-kan` package version;
- `efficient-kan` git commit and dirty status when using a source checkout;
- Python version;
- PyTorch version;
- CUDA availability and CUDA version;
- device and dtype;
- seeds;
- model dimensions and KAN hyperparameters;
- whether adaptive grid updates were used.

## Provenance Command

From a source checkout, write machine-readable provenance with:

```bash
python scripts/provenance.py --json
```

To force provenance for the current source checkout:

```bash
python scripts/provenance.py --json --source-checkout
```

From an installed package:

```bash
efficient-kan-provenance --json
```

If the installed command should report a specific source checkout:

```bash
efficient-kan-provenance --json --source-checkout --checkout-root /path/to/efficient-kan
```

The JSON includes:

- package name and resolved version;
- module version;
- installed distribution version when available;
- import mode and import path;
- source checkout root and `pyproject.toml` version when available;
- git commit, dirty status, and short status;
- Python version, implementation, and executable;
- platform information;
- PyTorch version, CUDA availability, and CUDA version.

## Example Provenance Artifact

Downstream projects can save the JSON directly:

```bash
python scripts/provenance.py --json --source-checkout > efficient_kan_provenance.json
```

A downstream manifest should include or reference that file. A minimal manifest
entry can look like:

```json
{
  "dependency": "efficient-kan",
  "package_version": "0.2.0",
  "git_commit": "75a6355621c21ce3fc1589a2da76e065361bc38d",
  "git_dirty": false,
  "import_path": "/path/to/env/site-packages/efficient_kan/__init__.py",
  "python_version": "3.12.11",
  "pytorch_version": "2.9.0+cu128",
  "cuda_available": true,
  "cuda_version": "12.8",
  "device": "cuda",
  "dtype": "float32",
  "seed": 123,
  "validation_command": "scripts/validate.sh"
}
```

Use actual values from the provenance command rather than copying this example.

## Recording Model State

When a trained model matters for reproducibility, save:

- the model `state_dict`;
- the architecture used to reconstruct the model;
- KAN constructor arguments;
- training objective and optimizer settings;
- random seed;
- grid-update schedule;
- package provenance JSON.

Example:

```python
import torch
from efficient_kan import KAN


model = KAN(
    [3, 16, 1],
    grid_size=5,
    spline_order=3,
    grid_eps=0.02,
    grid_range=(-1.0, 1.0),
)

torch.save(
    {
        "model_class": "efficient_kan.KAN",
        "layers_hidden": [3, 16, 1],
        "grid_size": 5,
        "spline_order": 3,
        "grid_eps": 0.02,
        "grid_range": (-1.0, 1.0),
        "state_dict": model.state_dict(),
    },
    "kan_model.pt",
)
```

The `grid` buffer is part of the state dict. If adaptive grid updates were used,
the saved state contains the adapted grids.

## Adaptive Grid Updates

Adaptive grid updates mutate model state. For reproducible research, record:

- whether `update_grid()` or `KAN.forward(update_grid=True)` was used;
- when updates occurred;
- which data split supplied the update batches;
- `margin`;
- `grid_eps`;
- whether updates occurred before evaluation.

Do not update grids on validation or test data unless the resulting state
mutation is an intentional and documented part of the estimator.

## Downstream Citation and Provenance Expectations

When downstream work depends on `efficient-kan`, cite or record:

- the package name, version, and repository URL;
- the exact commit SHA for source-checkout use;
- whether the work used a clean or dirty checkout;
- the provenance JSON emitted before running simulations or empirical analyses;
- the validation command used before generating final artifacts.

For papers, reports, or replication packages, include enough information for a
reader to identify the exact package state. A package version alone is not
enough when using an editable source checkout.

## Validation Before Evidence Runs

Before generating downstream research evidence from a new package state, run:

```bash
scripts/validate.sh
```

Then save:

```bash
python scripts/provenance.py --json --source-checkout
```

The validation gate is local. This project intentionally does not rely on
hosted CI/CD as the authoritative readiness signal.
